# mysql 常见错误

# 1. timestamp 默认值 '0000-00-00 00:00:00' 报错

# 问题描述

在创建表时，设置某一个时间列为 timestamp，并且设置默认值为 '0000-00-00 00:00:00'，会报一下错误信息：

[Err] 1067 - Invalid default value for {columnName}

# 问题源头 sql_mode，及解决

在 sql_mode 中 模式是不允许设置此默认值的，可以看一下 sql_mode 的具体信息，可以使用如下命令查看 当前会话或者全局的 sql_mode，如下所示：

SELECT @@GLOBAL.sql_mode; -- 查看全局模式 
SELECT @@SESSION.sql_mode; -- 查看当前会话的模式

可以看到下面的输出结果：

ONLY_FULL_GROUP_BY,STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_ENGINE_SUBSTITUTION

从上面我们可以看到 sql_mode 中有 NO_ZERO_IN_DATE 和 NO_ZERO_DATE

要解决这个问题可以吧 mode 中的 NO_ZERO_IN_DATE 和 NO_ZERO_DATE 设置去掉即可，可以这样设置

-- 其中mode的值为：'STRICT_TRANS_TABLES,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION'
SET GLOBAL sql_mode = {{mode}}; // 设置全局
SET SESSION sql_mode = {{mode}}; // 设置当前会话

# 常见 mode 说明

  1. ONLY_FULL_GROUP_BY：出现在select语句、HAVING条件和ORDER BY语句中的列，必须是GROUP BY的列或者依赖于GROUP BY列的函数列。
  2. NO_AUTO_VALUE_ON_ZERO：该值影响自增长列的插入。默认设置下，插入0或NULL代表生成下一个自增长值。如果用户希望插入的值为0，而该列又是自增长的，那么这个选项就有用了。
  3. STRICT_TRANS_TABLES：在该模式下，如果一个值不能插入到一个事务表中，则中断当前的操作，对非事务表不做限制
  4. NO_ZERO_IN_DATE：这个模式影响了是否允许日期中的月份和日包含0。如果开启此模式，2016-01-00是不允许的，但是0000-02-01是允许的。它实际的行为受到 strict mode是否开启的影响1。
  5. NO_ZERO_DATE：设置该值，mysql数据库不允许插入零日期。它实际的行为受到 strictmode 是否开启的影响。
  6. ERROR_FOR_DIVISION_BY_ZERO：在INSERT或UPDATE过程中，如果数据被零除，则产生错误而非警告。如果未给出该模式，那么数据被零除时MySQL返回NULL
  7. NO_AUTO_CREATE_USER：禁止GRANT创建密码为空的用户
  8. NO_ENGINE_SUBSTITUTION：如果需要的存储引擎被禁用或未编译，那么抛出错误。不设置此值时，用默认的存储引擎替代，并抛出一个异常
  9. PIPES_AS_CONCAT：将”||”视为字符串的连接操作符而非或运算符，这和Oracle数据库是一样的，也和字符串的拼接函数Concat相类似
  10. ANSI_QUOTES：启用ANSI_QUOTES后，不能用双引号来引用字符串，因为它被解释为识别符

# 2. Mysql连接本地报错：1130-host ... is not allowed to connect to this MySQL server

进入到数据库，将登陆的用户 host 字段修改为 %

update user set host = '%' where user ='root';

# 3. ERROR 1449 (HY000): The user specified as a definer ('mysql.infoschema'@'localhost') does not exist

进入到 mysql 命令行

  1. DROP USER 'mysql.infoschema'@'localhost';

  2. CREATE USER 'mysql.infoschema'@'localhost' IDENTIFIED BY 'password';

  3. GRANT SELECT ON . TO mysql.infoschema@localhost;

# 4. mysql监听的端口变成0

mysql 服务运行正常，但是3306端口却没有被占用，发现端口是0

<!-- 登录mysql，检查mysql 占用端口 -->
use mysql;
show global variables like 'port';
<!-- 查看配置文件的 skip-networking 是否配置，这个端口为 on会导致这个问题,进入到配置文件注释掉这个配置或者改为OFF即可  -->
show variables like 'skip_networking';

# mysql8 修改密码

  1. 修改配置文件，增加 skip-grant-tables 配置

  2. 免密登录 root 账号

  3. 修改密码,有两种修改方式

<!-- 方式1、查看当前认证方式以及用户，默认是 caching_sha2_password 认证 -->
select host, user, authentication_string, plugin from user;
<!-- 如果是默认认证方式，则修改为密码认证为 mysql_native_password -->
可以直接用sql修改：
update user set plugin = 'mysql_native_password' where user = 'root';
也可以在配置文件中添加下面的语句后重启：
default_authentication_plugin=mysql_native_password

<!-- 方式2、不能直接修改密码，会报错。要先清空root密码 -->
update user set authentication_string='' where user='root'; 
flush privileges;
<!-- 退出mysql, 删除/etc/my.cnf文件里的 skip-grant-tables ，重启 mysql 服务，再次登陆的时候是空密码登陆： -->
mysql -u root -p
<!-- 登陆后即可修改密码了： -->
ALTER USER 'root'@'%' IDENTIFIED BY '123456';
flush privileges;
