# git submodule 使用

# 删除子仓库

# 如果添加的子模块是一个空项目，没有任何文件，则会失败，然后再次进行添加子模块的操作(已经有文件，不是空项目)，依然失败，可以执行以下操作
# 执行以下命令，删除掉之前执行失败时，遗留的缓存
git rm --cached submodule
rm -rf submodule
rm -rf .git/modules/submodule

# 添加子仓库

# 再重新添加 submodule
git submodule add {{submoduleGitUrl}} {{dirPath}}

另外还有一种方式创建子模块

git add .gitmodules submodule
git commit -m "git submodule"
git submodule init
# 更新子模块内容
cd submodule && git submodule update

# 使用 git status 查看添加的子模块状态

 On branch master
    Changes to be committed:
    
        new file:   .gitmodules
        new file:   submodule

多了两个文件：.gitmodules, submodule, 其中 .gitmodules 内容包含 submodule 的主要信息，指定子模块 reposirory 路径,以及本地子模块路径, submodule 文件夹就包含了子模块的文件

# 更新子模块

# 进入到子项目更新
cd submodule && git pull
# 也可以直接在父项目中更新
git submodule foreach git pull

在 superProject 运行 git pull 会将子项目的 header 指向父项目中记录的 commit id。 所以如果先在子项目中运行了 git pull，再去父项目执行 git pull，可能会导致代码版本不一致 注意更新 submodule 的时候如果有新的 commit id 产生，需要在父项目产生一个新的提交，superProject 文件中的 submodule commit会变为最新的commit id。

# 提交 submodule

# 子模块提交内容和普通仓库一致
git add fileName
git commit -m'msg'
git push
# 提交完毕子模块内容后，子模块 header 指向的 commit id 发生变化，还需要更新 superProject 的内容
# 这时候在 superProject 中会提示 submodule 有修改，需要提交

# 删除 submodule

git rm --cached pod-library
rm -rf pod-library
rm .gitmodules
# 更新 .git/config submodule 相关内容
# 删除 .git/modules/submoduleName 文件
# 提交到远程服务器
git commit -a -m 'remove submodule'
