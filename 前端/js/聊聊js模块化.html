  • IIFE

javascript 作为一门脚本语言，在设计之初主要是用来执行独立的脚本任务。随着项目的复杂化，我们通常需要执行大量的javascript 代码来实现网页交互。为了将 JavaScript 程序拆分为可按需导入的单独模块，这其中涌现了一些模块化定义以及其实现。其中主要就包括 IIFE, AMD, CMD, CommonJS, UMD, ESM 等。

# IIFE

IIFE（Immediately Invokable Function Expressions 立即执行函数）

特点：

  1. 解决了变量保护，全局污染的问题
  2. 不能有效的解决模块间依赖

它通常是这样，不需要通过第三方实现

(function () {
 // code
})()

# AMD

AMD（Asynchronous Module Definition）他是一种异步模块定义方式，第三方库 requirejs 实现了这种模块机制。

该机制规定需要去实现 define 和 require 两个全局变量。其中 define 用于定义模块，它接收三个参数

  1. 第一个字符串类型的可选定义的模块名称，如果不传则认为是匿名模块。第二个参数
  2. 第二个参数为该模块所依赖的模块列表，加载定义的模块时，会优先去加载他所依赖的模块
  3. 第三个参数可以是一个方法，或者是对象。如果是一个方法则他会被执行一次，用来创建模块对象。如果是一个对象则这个对象直接被作为模块的导出值

特点：

  1. 依赖必须事先声明，加载完成后，再去执行代码
  2. 模块异步加载，指定回掉函数
// 定义模块
define("alpha", ["require", "exports", "beta"], function (require, exports, beta) {
  // 加载模块
  var a = require('a')
  exports.verb = function() {}
})

# Commonjs

主要是 javascript 在 nodejs 端的模块机制，其中在浏览器端 webpack 和 browserify 也有实现。它主要实现了 module 这个全局变量。并且在 module 变量里还有 exports, require, filename 给到内部代码使用

特点：

  1. 同步加载，资源加载完成后再去执行代码
  2. 每个文件都是一个模块
  3. 第一次加载后会在 require.cache 中缓存起来，所以每次执行 require 加载同一个模块，得到的都是同一个对象
  4. nodejs 端模块加载是运行时同步加载，直接读本地磁盘，比较快速
  5. 浏览器端通过打包编译时加载

它的模块通常是这样的：

const fs = require('fs')
const { PI } = Math
// 导出 area 变量
exports.area = (r) => PI * r ** 2
// 导出 circumference 变量
exports.circumference = (r) => 2 * PI * r

实际上在每个文件代码执行之前，nodejs 会用下面的函数对其进行封装

(function(exports, require, module, __filename, __dirname) {
// 模块代码实际存在于此处
})

简单实现

(function (modules) {
    // 用来缓存加载过的模块
    const moduleCache = {}
    // 加载模块方法
    function require(path) {
       // 已经加载过该模块 直接返回
       if (moduleCache[path]) return moduleCache[path].exports
        // 加载模块后 需要生成一个 module 变量
        const module = moduleCache[path] = {
            moduleName: path,
            exports: {}
        }
        modules[path].call(modules.exports, module.exports, require, module)
        return module.exports
    }
    return require("entry")
}({
    a: function (exports, require, module) {
        // code
    },
    b: function (exports, require, module) {
        // code
    },
    entry: function (exports, require, module) {
        // code
    }
}))

# UMD

为了统一 Commonjs 和 AMD 生态系统，通用模块定义（Universal Module Definition）规范应运而生，本质上 UMD 定义的模块会在启动时检测要使用哪个模块系统，然后进行适当配置，并把所有逻辑包装在一个立即调用的函数表达式(IIFE)中

所有 umd 模块化的代码在外层应该都有下面的这部分代码，用来适配不同的环境

(function (global, factory) {
    // 判断环境
    if (typeof exports === 'object' && typeof module !== 'undefined') {
        // 非严格模式的 Commonjs
        factory(exports)
    } else if (typeof define === 'function' && define.amd) {
        // AMD 模块化形式，注册为匿名模块
        define(['exports'], factory)
    } else {
        if (typeof globalThis !== 'undefined') {
            global = globalThis
        } else {
            global = globalThis || self
        }
        factory.call(global, global[exportName] = {})
    }
}(this, function (exports) {}))

# ESM

作为 javascript 模块化的集大成者，它可以使用在浏览器端和服务端，不过由于历史原因都需要使用 babel 去进行编译，或者添加 --experimental-modules

特点：

  1. 使用 import 引入，并且需要提前在顶层声明好依赖
  2. 支持 tree-shaking
  3. 输出的是值引用，加载不会缓存
  4. 编译时确定依赖关系，编译时加载

它的模块通常是这样的：

import React, { useEffect, useState } from 'react'
export const config = {}
export default function Foo() {}

# npm

经过多年的发展，现阶段使用较多的模块化机制为：浏览器端项目：ESM， nodejs 端：Commonjs，以及针对 Library 的：UMD。通常我们现在在开始一个前端项目的时候，都会使用 npm init 去进行初始化，那么这个与今天的主题js模块化有什么关系呢。首先 npm init 会生成一个 package.json 文件，里面有几个与模块化相关的字段：main，module，browser

这三个字段所表示的都是该模块的入口文件，即其他模块在加载该模块时由他们去决定去哪个文件找到导出的对象，但是这三个字段又分别有针对环境及优先级

环境：

  1. main 浏览器端和 nodejs 端
  2. module 浏览器端和 nodejs 端
  3. browser 只在浏览器端有效

优先级方面这里我们采用常用的打包工具（webpack）和模块化形式来做对比，webpack 通过设置不同的 target 来切换不同的环境

  1. Webpack + web + ESM：这是现代化前端项目最常见的场景，经测试优先级为：browser > module > main
  2. Webpack + web + Commonjs：采用 require 的形式去导入模块，经测试优先级为：browser > module > main
  3. Webpack + node + ESM：nodejs 端采用 ESM，并且使用 webpack 打包，经测试优先级为：module > main
  4. Webpack + node + Commonjs：nodejs 端采用 ESM，并且使用 webpack 打包，经测试优先级为：module > main
  5. nodejs + Commonjs：比较常规的 nodejs 项目，经测试优先级为：只有 main 字段生效
  6. nodejs + ESM: 采用 node index.mjs --experimental-modules 的形式，可以让 nodejs 支持 ESM，经测试优先级为：只有 main 字段生效

# 总结

  1. 如果模块需要兼容浏览器端和服务器端，最好同时指定 module 和 main 字段，因为 module 是 ESM 的，在现代化的打包工具支持下能够很好的支持 tree-shaking 能力，尽可能的去优化打包输出
  2. 如果我们要创建一个 library 并且要兼容浏览器端和 nodejs 端，最好同时输出 ESM, Commonjs, UMD 三种模式的包
  3. 如果适用侧没有打包工具，并且要支持浏览器和 nodejs 端，则指定 browser 和 main
  4. 如果只在 nodejs 端使用，则指定 main 字段
