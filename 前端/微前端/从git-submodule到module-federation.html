# 1. git submodule 是什么

它实际上只是标准的 git repository，在某些场景下项目需要包含并使用另一个项目。 也许是第三方库，或者你独立开发的，用于多个父项目的库。这时候它就派上用场了。它能够帮我们统一管理一些公用的代码，并且提供给父项目使用。他很健壮，被使用在大量的项目中，比如现在比较流行的 lerna + npm packages + git submodule 前端项目组织结构。

现在组内的各大项目普遍使用了微前端技术，各个应用都会使用一些公用的组件库，工具函数等，这就需要在统一的地方去维护，通常我们在这里使用的是 git submodule 方法去管理组件库源代码。不过这种方案也有一些弊端。这里我们来讨论下如何去规避，解决这些问题。

在微前端中使用 git submodule 遭遇的一些痛点：

1. 需要一定的学习成本，对于git不太熟悉的人使用起来较为困难
2. 它存储的是源代码，不会独立部署，当有组件更新时需要将依赖该组件的所有项目都进行重新部署
3. 某些场景下一个项目可能需要使用另一个项目中的组件时，需要花费比较大的精力去将组件迁移到公共的组件库，但是也并不能保证这次迁移是完全安全的
4. 如果不去进行迁移那么通常的做法是直接 copy 一份组件的源代码到公共组件库，这样做导致同一个组件存在多个不同版本，后期维护起来会相当困难。

虽然 git submodule 足够的强大，但是也不一定能够完美的适配微前端架构，这里介绍一种 module federation 方案。

# 2. module federation 是什么

首先他是 webpack@5.x 中提出的概念，从官方文档来看，它将我们的每一个项目构建都当作一个容器，这个容器能够使用其他容器所暴露出来的模块，同样反过来也能够暴露模块给到其他容器使用。它对外暴露的模块实际上是一个个编译后的 js 文件，所以我们在代码中就能够直接引入使用。

它提供了以下功能能够供我们解决上面的一些问题：

  1. 配置简单，只需要在webpack 中增加对应的插件，并且暴露或者引入所需的模块
  2. 每一个项目能够独立部署，对外暴露一些编译好的 js 等文件，使用侧通过部署的路径去访问引用所需的模块文件
  3. 每个项目容器都能够向外暴露或使用其他项目所暴露的模块，所以上面我们遇到的项目交叉引用的问题也就迎刃而解

# 3.2 在目标项目中的 vue.config.js 中添加 module federation 配置

// 模块提供方配置
const packageName = require('./package.json').name;
module.exports = {
  ...,
  chainWebpack: config => {
    // 由于 webpack splitchunk 与 module federation 冲突
    // 组件资源提供方也就是 remote 放需要添加下面这行设置
    config.optimization.delete('splitChunks')
    config
      .plugin('module-federation-plugin')
      .use(require('webpack').container.ModuleFederationPlugin, [{
        name: packageName,
        filename: `${packageName}.js`,
        library: {
          // 详情查看：https://github.com/umijs/qiankun/issues/1148
          // 在使用 qiankun 微前端的架构下
          // 如果使用 var 需要弃用 qiankun 得沙箱机制，需要设置
            // start({
            //   sandbox: false
            //   sandbox: { strictStyleIsolation: false },
            // });
          // 或者将 type 设置为 window，即可不对 qiankun 做调整
          type: 'window',
          name: packageName
        },
        exposes: {
          './Btn': './src/components/btn/index',
          './Content': './src/components/content/index',
        },
        // 使用共享依赖配置可能会出现 Shared module is not available for eager consumption 的错误
        // 参考官方文档：https://webpack.js.org/concepts/module-federation/#uncaught-error-shared-module-is-not-available-for-eager-consumption
        // 因为乾坤需要在入口文件暴露 mounted, unmounted 等生命钩子，所以这里并不能使用官网给出的这套解决方案
        // shared: {
        //   moment: {
        //     // singleton: true
        //   },
        //   vue: {
        //     // singleton: true
        //   },
        // }
      }]
    )
  }
}

# 3.3 一些不足或缺陷解决

  1. 升级过程中发现使用的微前端框架和 webpack@5.x 存在不兼容的问题，qiankun 会报不能加载子应用的错误。所以这里的解决方案是采用自定义一个 html plugin 插件去解决
  2. 在提供公共模块侧的配置中，不能配置共享依赖，可能会出现这个错误 Shared module is not available for eager consumption 的错误，详情见官方文档：https://webpack.js.org/concepts/module-federation/#uncaught-error-shared-module-is-not-available-for-eager-consumption。由于乾坤需要在入口文件暴露mounted, unmounted 等生命钩子，所以这里并不能使用官网给出的这套解决方案。
// 自定义一个 html plugin 解决 qiankun 和 webpack@5.x 不兼容的问题
/**
 * 通过html-webpack-plugin暴露的钩子处理数据
 */
 class HtmlWebpackCustomPlugin {
    constructor(htmlWebpackPlugin, options) {
        this.htmlWebpackPlugin = htmlWebpackPlugin;
        // 外部传入配置
        this.options = options || {};
    }

    apply(compiler) {
        // 插件名
        const pluginName = 'HtmlWebpackCustomPlugin';
        if (compiler.hooks) {
            compiler.hooks.compilation.tap(pluginName, compilation => {
                const hook = this.htmlWebpackPlugin.getHooks(compilation);
                hook.alterAssetTags.tapAsync(pluginName, (htmlPluginData, callback) => {
                    const cloneHtmlPluginData = { ...htmlPluginData };
                    const entryScript = cloneHtmlPluginData.assetTags.scripts.find(item => {
                        //
                        return item.attributes.src.includes('app.js');
                    });
                    if (entryScript) {
                        entryScript.attributes.entry = true;
                    }
                    callback(null, cloneHtmlPluginData);
                });
            });
        }
    }
}

module.exports = HtmlWebpackCustomPlugin;

// qiankun 子应用的 vue.config.js 配置都需要添加自定义的 html plugin
const htmlWebpackPlugin = require('html-webpack-plugin');
const MyAwesomeWebpackPlugin = require('../plugin/html-plugin')
module.exports = {
  ...,
  configureWebpack: config => {
    config.plugins.push(new MyAwesomeWebpackPlugin(htmlWebpackPlugin))
  }
}

综上可以看出 module federation 也并非完美的解决方案，不过相比起 git submodule 的一些问题我觉得这些都不算太严重，至少能解决并且对业务功能不会造成太大的负担。至于 git submodule 和 module federation 怎么去抉择那就仁者见仁智者见智了。
